# Website Technologies Scraper

## Getting Started

To run this project locally, make sure you have Node.js v22 or above installed on your machine.

### Dependencies

This project uses the following packages:

- `wappalyzer` — headless browser-based technology detection
- `parquetjs-lite` — reading the `.parquet` input file

### Installation

Clone the repository and install the dependencies:

```bash
npm install
```

### Execution

To start the scraper, run:

```bash
node app.js
```

When it finishes, the script will create an `output.json` file in the root folder with the scanned domains, the technologies found, and the proof of detection. A sample output file `max-output.json` is included in the repository. It was generated by scanning 200 domains and found 343 unique technologies — it's the best run I got during testing, so I kept it as a reference example of what the output looks like.


You can configure the scraper by editing the parameters in `app.js`:

```javascript
websiteTechnologiesScraper('./input_data.parquet', 5, 30000);
//                                                 ^    ^
//                                       packetSize    timeout (ms)
```

- `packetSize` — how many domains are scanned at the same time. The default is `5`. You can increase it to speed things up, but the higher the number, the more likely it is that some domains will timeout or get skipped entirely, which means you could miss technologies. If you notice a lot of empty results, try lowering this number.
- `timeOut` — how long the scraper waits for a site to load before giving up, in milliseconds. The default is `30000` (30 seconds). Increase it if you are scanning slow websites.

## Solution Explanation

My approach had two phases.

**Phase 1 — Custom scraper**

I started by building a basic scraper using the fetch API and cheerio. The idea was to look through the raw HTML of each website (things like script `src` attributes, `<meta name="generator">` tags) and check the HTTP response headers (like `Server` or `X-Powered-By`) to figure out what technologies a site is using. This works fine for simple websites, but it breaks on modern apps built with React because those pages start with almost no HTML at all — the content is generated by JavaScript after the page loads, so there is nothing useful to parse.

**Phase 2 — Wappalyzer integration**

To fix that, I switched to wappalyzer. Wappalyzer opens a real headless browser, runs the JavaScript on the page, and checks the result against a huge list of known technology patterns. This way it can detect technologies even on complex modern websites.

The final implementation follows this flow:

1. Read the `.parquet` input file using `parquetjs-lite` and extract all domains into an array
2. Split that array into packets of 5 domains each using a `packetArray` helper function
3. Loop through the packets one by one — for each packet, map every domain to a `wappalyzer.open()` call and wrap it in a Promise
4. Resolve all 5 Promises in parallel using `Promise.all`, so the domains in a packet are scanned at the same time
5. For each scanned domain, collect the detected technologies and store them in the results object
6. Add every found technology to a JavaScript `Set` to automatically filter out duplicates across different domains
7. After all packets are done, write the final results to `output.json`

**The proof**

Wappalyzer does not just say if a technology is present or not. For each detected technology it assigns a confidence score from 0% to 100%, based on how many patterns matched and how reliable those patterns are. For example, if a technology is detected through multiple signals (a script tag, a response header, and a global JS variable all pointing to the same thing), the confidence will be 100%. If only one weak signal matched, it might be 50% or lower. This is more useful because it lets you filter out false positives and focus on the results you can actually trust.

## Debate Topics

### Main issues with the current implementation and how to fix them

When I tested the script on a faster machine, I ran into this warning: `MaxListenersExceededWarning: Possible EventEmitter memory leak detected`.

What happened is that `Promise.all` was trying to open too many headless browser instances at the same time, which overloaded the Node.js event system.

To fix this I set a packet size (`PACKET_SIZE = 5`) so the script only processes 5 domains at a time instead of all at once. I also increased the event listener limit with `process.setMaxListeners` to stop the warning from showing up. And I wrapped everything in a `try...catch` so if one domain fails or times out, the rest of the domains keep running normally.

### How would you scale this solution for millions of domains in 1-2 months?

For this scale, I would use the BuiltWith API instead of running a local scraper. BuiltWith is a service that already has technology data for hundreds of millions of websites. Instead of visiting each domain ourselves, we just send the domain to their API and get back the full list of technologies it uses.

The way it would work in practice: we take the full list of domains, split it into chunks, and send each chunk to the BuiltWith API endpoint. For each domain they return a JSON response with all the detected technologies, the categories they belong to, and when they were last seen. We collect all those responses and save them to a database or a file. Since we are not running any browser or scraping anything ourselves, we can process thousands of domains per minute depending on our API plan. The main tradeoff is cost, since BuiltWith is a paid service, but for millions of domains it is much faster and more reliable than running our own infrastructure.

### How would you discover new technologies in the future?

If we keep using Wappalyzer, its list of technology patterns is open source and people keep adding to it. Updating the npm package regularly is enough to stay current.

But if we want to find technologies that no tool knows about yet, we can bring back the basic scraper from phase one. The idea is to collect all the unusual or unknown values from HTTP headers and global JavaScript variables across thousands of sites, then look for patterns that keep showing up. If the same unknown header appears on hundreds of sites, it is probably a technology worth adding to the list.
